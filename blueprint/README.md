# AINTLIB Blueprint

This project's mathematical blueprint is authored with
[`leanprover/verso-blueprint`](https://github.com/leanprover/verso-blueprint)
(the Verso-based tool behind verso-flt, verso-sphere-packing, verso-carleson).
Each declaration is paired with its mathematical statement and a paragraph-level
proof sketch; the generator renders an interactive HTML dependency graph and a
progress summary. Lean status (sorry-free vs. in-progress) is auto-computed from
the `(lean := …)` references — there is no `\leanok` to keep in sync.

## Build and render

One command builds the Lean library and renders the site:

```bash
./scripts/ci-pages.sh
```

That script runs exactly:

```bash
lake build AINTLIB
lake env lean --run AINTLIBMain.lean --output _out/site
```

(then asserts the key output files exist). It avoids compiling the
`blueprint-gen` executable and its native artifacts, which matters for
Mathlib-heavy projects and cold CI. A compiled executable is also available for
repeated local runs:

```bash
lake exe blueprint-gen --output _out/site
```

## Output

- HTML site root (what GitHub Pages serves): **`_out/site/html-multi/`**
  - `index.html` — blueprint home
  - `Dependency-Graph/` — interactive `{blueprint_graph}` page
  - `Blueprint-Summary/` — `{blueprint_summary}` progress page
  - per-chapter pages (e.g. `Addition/`, `Multiplication/`, `Collatz/`)
  - `-verso-data/blueprint-preview-manifest.json` — node/edge + status manifest

`_out/` is gitignored and is regenerated by the command above.

## Source layout

- `AINTLIB/Blueprint.lean` — top-level blueprint document; `{include …}`s each
  chapter and hosts the `{blueprint_graph}` / `{blueprint_summary}` directives.
- `AINTLIB/Chapters/` — one Verso module per chapter (currently the starter
  Addition / Multiplication / Collatz examples — to be replaced by the real
  number-theory chapters in Task 1.3).
- `AINTLIB/Core.lean` — forces the cited mathlib declarations
  (`import Mathlib.NumberTheory.…`) into the build so `(lean := "Mathlib.…")`
  references in the blueprint resolve and are auto-tracked as sorry-free.
- `AINTLIB.lean` — library root; imports both `AINTLIB.Blueprint` and
  `AINTLIB.Core`.
- `AINTLIBMain.lean` — generator entry point (`root := AINTLIBMain` /
  `blueprint-gen` exe in `lakefile.lean`).
- `scripts/ci-pages.sh` — the build-and-render check the Pages workflow runs.
- `.github/workflows/` — `pages.yml` (caller) + `blueprint-pages.yml`
  (reusable build/deploy). Deployment is reconciled in a later task; harmless
  with no remote configured.

## Pinned versions

A lake project shares ONE toolchain across all dependencies. verso-blueprint
targets v4.30.0, so AINTLIB is pinned to the v4.30.0-rc2 line (down from the
v4.31.0-rc1 of Phase 1.1). This is the proven-compatible set from `verso-flt`
(mathlib + verso-blueprint).

| Component         | Pin                                                    |
| ----------------- | ------------------------------------------------------ |
| Lean toolchain    | `leanprover/lean4:v4.30.0-rc2`                         |
| mathlib           | `229580e5e4f991a61279c96e55b6c45c88866718`            |
| VersoBlueprint    | `v4.30.0` (rev `b37defe8f6fb99df8abfaa5833004776c04f5a1a`) |
| verso             | `v4.30.0-rc2` (rev `0bce2769d753e69fe092f4f2b02cb1428d6287a6`) |

The full transitive dependency set (verso, subverso, aesop, batteries, Qq, Cli,
importGraph, proofwidgets, plausible, LeanSearchClient, MD4Lean, checkdecls) is
frozen in `lake-manifest.json`, adapted from verso-flt's committed manifest. To
populate mathlib's prebuilt `.olean` cache before building:

```bash
lake exe cache get
```
